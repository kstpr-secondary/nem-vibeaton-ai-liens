# GLSL Patterns

Reusable GLSL shader patterns for the Vibeaton AI Liens rendering pipeline. All shaders use **sokol-shdc annotations** (`@vs`, `@fs`, `@block`, `@program`, `@ctype`) and target **OpenGL 3.3 Core** via `--slang glsl330`.

This skill covers **GLSL code, sokol-shdc dialect quirks, uniform block layout, and shader error handling**. Concrete shader implementations are in separate reference files loaded on demand:

| Reference | Open when |
|-----------|-----------|
| `references/glsl-patterns-lighting.md` | Authoring unlit, Lambertian, Blinn-Phong, or normal-mapped shaders |
| `references/glsl-patterns-special-shaders.md` | Authoring skybox, procedural sky, or line-quad shaders |
| `references/glsl-patterns-game-fx.md` | Authoring game FX (plasma beam, explosion, shield, laser) fragment shaders |
| `references/glsl-patterns-custom-shaders.md` | Implementing the custom shader hook (R-M5), extending VFX, or wiring game-workstream shaders |

**Companion skills:** `sokol-shdc/SKILL.md` (annotation dialect, CLI, CMake, generated headers), `sokol-api/` (sokol_gfx pipeline plumbing). AGENTS.md global rules always apply.

## 1. sokol-shdc Annotation Dialect

All GLSL files are annotated for `sokol-shdc` translation into C headers. The annotation syntax is:

```glsl
@vs <name>                   // vertex shader entry point
uniform <block_name> { ... }; // uniform block (becomes C struct)
in vec3 position;             // vertex attribute input
out vec3 world_pos;           // VS-to-FS semantic output
void main() { ... }
@end

@fs <name>                   // fragment shader entry point
uniform <block_name> { ... };
in vec3 world_pos;           // FS semantic input (matches VS out)
out vec4 frag_color;         // framebuffer output
void main() { ... }
@end

@program <name> <vs_name> <fs_name>  // links VS + FS into a program
```

### Block includes

Use `@block` to define reusable uniform groups and `@include_block` to inject them:

```glsl
@block common_fs
uniform scene_params {
    vec3 light_dir;
    vec3 light_color;
    float light_intensity;
};
@end

@fs custom_fs
@include_block common_fs
uniform custom_params { ... };
void main() { ... }
@end
```

### Type mapping with @ctype

Use `@ctype` to map GLSL types to C/C++ types in the generated header:

```glsl
@ctype mat4 glm::mat4
@ctype vec4 glm::vec4
@ctype vec2 glm::vec2
```

This ensures the generated C struct matches GLM types. Only add `@ctype` for compound types — `float` and `int` map natively and must not have a `@ctype` entry (`glm::float` does not exist).

**Warning — do not add `@ctype vec3 glm::vec3`.** `glm::vec3` is 12 bytes but std140 pads every `vec3` to 16 bytes. If you use `glm::vec3` in a C uniform struct, the layout will silently diverge from the shader, producing wrong colors or black geometry. **Use `vec4` in all uniform block fields and swizzle `.xyz` in shader code.** See §2.

### sampler declarations

Textures and samplers must be declared as separate uniform blocks:

```glsl
uniform texture2D albedo_tex;
uniform sampler smp;
```

In C++, bind via `sg_bindings` using the slot names generated by sokol-shdc (e.g., `SLOT_albedo_tex`). The prefix is `SLOT_`, not `SG_SLOT_` — using the wrong prefix is a compilation error.

## 2. Uniform Block Layout Rules

### std140 alignment table

OpenGL std140 requires specific alignment for uniform blocks:

| GLSL Type | Size (bytes) | Alignment (bytes) |
|-----------|-------------|-------------------|
| `float` | 4 | 4 |
| `vec2` | 8 | 8 |
| `vec3` | 12 | 16 |
| `vec4` | 16 | 16 |
| `mat2` | 32 (2x16) | 16 |
| `mat3` | 48 (3x16) | 16 |
| `mat4` | 64 (4x16) | 16 |

**Rule:** Never put a `float` or `vec3` after a `vec4` in the same block without padding — std140 pads the preceding field to 16 bytes, so the next field starts at offset 16. The C struct from sokol-shdc must match exactly.

**Best practice:** Let sokol-shdc generate the C struct via `@ctype`. Never hand-pad the C struct — mismatches produce garbled uniforms (black screen or wrong colors).

### Safe block ordering pattern

```glsl
// GOOD: pack all vec3 quantities into vec4 fields; scalar data packs after vec4 without gaps
uniform fs_params {
    vec4 base_color;           // offset 0   — use .rgba
    vec4 light_dir_ws;         // offset 16  — use .xyz; .w = unused
    vec4 light_color_inten;    // offset 32  — use .xyz for color, .w for intensity
};

// BAD: vec3 in uniform block — GLSL std140 pads vec3 to 16 bytes but glm::vec3 is only
// 12 bytes, so the C struct layout will NOT match the shader. Silent data corruption.
// Do NOT do this:
uniform bad_params {
    vec3 light_dir;       // GLSL: offset 0, 16 bytes with padding
    float intensity;      // GLSL: offset 16; but C glm::vec3 puts this at offset 12 → MISMATCH
};
```

## 3. Shader Error Handling Pattern

When a custom game-workstream shader fails to compile:

1. The renderer detects failure: `sg_query_shader_state(shader) == SG_RESOURCESTATE_FAILED`.
2. Logs via the custom logger registered in `sg_setup()`. There is no `sg_report_error()` function.
3. Creates and returns the magenta fallback pipeline for that material.
4. The object renders as solid magenta — visible but obviously broken.

**Critical:** Never propagate shader compilation errors to the game workstream. The renderer owns the fallback contract. Custom shaders are best-effort.

## 4. Quick Reference: Lighting Formulas

| Model | Formula | Notes |
|-------|---------|-------|
| **Unlit** | `color` | No lighting math |
| **Lambertian** | `max(N.L, 0) x albedo x light_color x intensity + albedo x 0.15` | Diffuse only |
| **Blinn-Phong** | `N.L term + pow(max(N.H, 0), shininess) x spec x 0.5 + ambient` | H = normalize(L + V) |
| **Fresnel rim** | `(1 - N.V)^exponent x color` | Shield VFX style |

**Common vectors:**
- `N`: surface normal (world space, normalized)
- `L`: light direction vector pointing **toward** the light (normalized)
- `V`: view direction = normalize(camera_pos - world_pos) (normalized)
- `H`: half vector = normalize(L + V) (normalized)

## 5. Depth and Alpha State Reference

| Pipeline type | depth compare | depth write | cull mode | blend state | Draw order |
|---------------|--------------|-------------|-----------|-------------|------------|
| Opaque (unlit, Lambertian, Phong) | `LESS_EQUAL` | `true` | `BACK` | none | any |
| Skybox | `LESS_EQUAL` | `false` | `NONE` | none | **first** |
| Alpha-blended (line quads, VFX) | `LESS_EQUAL` | `false` | `NONE` | `SRC_ALPHA` / `ONE_MINUS_SRC_ALPHA` | back-to-front |
| Error/fallback pipeline | `LESS_EQUAL` | `true` | `BACK` | none | same as opaque |

**Discard pattern in fragment shaders:**
```glsl
if (frag_color.a < 0.01) discard;
```

## 7. Vertex Shader Template (shared across all pipelines)

Every vertex shader passes these outputs to the fragment shader:

```glsl
out vec3 world_pos;       // world-space position
out vec3 world_normal;    // world-space normal (transformed by normal_mat, normalized)
out vec2 uv;              // texture coordinates
```

**Normal matrix is computed on the CPU** and uploaded as a `mat4` (`glm::mat4(glm::transpose(glm::inverse(glm::mat3(model))))`). In the shader: `world_normal = normalize(mat3(normal_mat) * normal)`. Never compute `transpose(inverse(...))` inside the vertex shader — this contradicts the renderer-specialist decision (see renderer-specialist SKILL §7) and wastes GPU cycles. Packing the mat3 in a mat4 avoids the std140 column-padding mismatch that `glm::mat3` would introduce.
