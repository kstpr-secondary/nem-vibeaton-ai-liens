## Expected Outcome

Running `renderer_app` shows a "Shadow Demo Controls" ImGui window. Enabling "Shadow Demo mode" replaces the existing diverse scene with a focused shadow validation scene: a ground plane, a tall post, a floating slab, a shadow-receiving sphere, a comparison (non-shadow-receiving) BlinnPhong sphere, and a transparent sphere, all illuminated by a fixed directional light slightly above horizontal. `BlinnPhongShadowed` objects receive visible hard shadows with no acne. The comparison `BlinnPhong` sphere has no shadow darkening. The transparent sphere is unaffected. The orbiting camera shows spatially stable shadows with no per-frame crawl. Enabling "Show shadow map" adds a greyscale corner-quad overlay in the bottom-right showing shadow map depth contents.

## Fallbacks

- **T-2 (shadow_debug_sampler compare field)**: If sokol's GL backend requires `compare = SG_COMPAREFUNC_NEVER` to disable comparison sampling, and the generated sampler produces validation warnings, try leaving the compare field at its zero-init default. If GL still interprets the depth texture through comparison mode with the debug sampler, fall back to manually linearizing depth in the FS: `float d = texture(sampler2D(shadow_tex, debug_smp), v_uv).r; frag_color = vec4(d, d, d, 1.0);` is already doing raw reads — the issue is the sampler type, not the shader. If a non-comparison sampler cannot sample a depth attachment, use a separate non-depth color attachment for debug (renders depth as R32F color). Escalate to human if neither works.
- **T-2 (shadow_debug.glsl binding name collision)**: The generated sokol-shdc constants `VIEW_shadow_debug_shadow_tex` and `SMP_shadow_debug_debug_smp` follow the pattern `VIEW_<program>_<name>`. If these names differ (check the generated `.glsl.h`), use the actual generated constants — never hardcode integer indices.
- **T-1 (shadow ortho quality)**: If shadows appear visibly pixelated at the demo scene scale after T-2's constant change to 25.0f half-size, reduce further to 15.0f. A 15.0f half-size on a 2048px map gives texels ≈ 0.015 world units — more than sufficient for a 20-unit ground plane.
- **T-3 (sokol_gfx.h include in main.cpp)**: If including `shadow_pass.h` from `main.cpp` causes compilation issues (duplicate symbol or platform include conflicts), extract only the three needed declarations (`shadow_debug_set_visible`, `shadow_debug_is_visible`, `shadow_debug_draw`) into a minimal `shadow_debug_api.h` that does NOT include `sokol_gfx.h`. Move `sg_pipeline` parameter of `shadow_debug_draw` behind a forward declaration or a `void*` cast if the type is unavailable. As a last resort: add `renderer_set_shadow_debug_visible(bool)` wrapper to the renderer (requires interface bump GATE — escalate to human before proceeding).

## Human Checkpoint

**Run**: `cmake --build build --target renderer_app renderer_tests && ./build/renderer_tests && ./build/renderer_app`

**Look for**:
1. Build exits 0 with no new warnings. `renderer_tests` exits 0 (9 tests still pass).
2. Application starts. Console shows: shadow pass init completed (handle non-zero), shadow depth pipeline created, shadow_debug pipeline created without sokol validation error, `shadow_debug_init` completed (VBO and IBO handles non-zero).
3. "Shadow Demo Controls" ImGui window is visible. Enable "Shadow Demo mode" checkbox.
4. With shadow demo active:
   a. Scene switches to demo scene with an orbiting camera looking at a ground plane, tall post, floating slab, two spheres, and one semi-transparent sphere.
   b. At least one clear shadow boundary is visible on the ground plane (shadow from the post or floating slab). The receiver sphere (blue, BlinnPhongShadowed) shows shadow darkening on the face away from the light.
   c. No shadow acne (self-shadowing noise/stripe artifacts) on any illuminated face.
   d. Comparison sphere (yellow, BlinnPhong) shows NO shadow darkening — illuminated uniformly on the lit side, no darker patches.
   e. Transparent sphere renders with correct semi-transparency, no shadow participation visible.
   f. After 30+ seconds of orbiting: shadows remain spatially stable — no per-frame shadow crawl, jitter, or severe flickering.
5. Enable "Show shadow map" checkbox. A greyscale quad appears in the bottom-right corner. The quad shows some depth variation — not perfectly uniform grey — confirming scene geometry is present in the shadow map.
6. Disable "Show shadow map". Quad disappears. No artifacts. No sokol validation errors on toggle.
7. Disable "Shadow Demo mode". Existing diverse scene renders identically to before.

**Pass**: All 7 criteria met.

**Stop**: Any of — build exits non-zero; `renderer_tests` exits non-zero; shadow demo scene doesn't appear when checkbox is enabled; no visible shadow on ground plane or receiver sphere; acne stripes visible on lit faces; comparison sphere has visible shadow darkening; transparent sphere has unexpected shadow artifacts; corner quad does not appear when "Show shadow map" is enabled; sokol validation errors during normal operation.
